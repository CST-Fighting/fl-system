# 算力任务需求管理数据库设计

## 1. 文档目标

本文档面向当前 `src/pages/ComputingSchedule/TaskManagement/index.tsx` 页面，以及后续真实后端接入需求，给出一套可落地的数据库设计方案。

目标不是做"前端随机 mock"，而是建设一套：

- 能存拟真数据
- 能接入后端服务
- 能支撑当前前端正常展示
- 后续能平滑替换为真实生产数据
- 能继续扩展任务调度、资源监控、告警等能力

本文档重点覆盖：

- 页面功能到数据库实体的映射
- 各类表设计（任务需求、节点资源、监控时序、告警、地图分布）
- 后端 API 如何从数据库取数
- 拟真数据如何流入数据库
- 未来真实数据如何并入

基础表结构参考：《联邦学习算力网络平台 — 数据库建表文档》（`data/数据库建表文档.md`），本文档在其基础上扩展与细化。

## 2. 当前功能界面需要支撑的核心能力

当前算力任务需求管理页面包含以下能力：

1. 顶部统计概览（任务总数、运行中任务、空闲节点、资源利用率）
2. 算力需求分层管理视图（按高/中/低优先级分组展示任务需求）
3. 算力中心能力视图（CPU/GPU/存储/网络资源占比环形饼图）
4. 全国算力节点分布（中国地图散点图，展示 26 个算力中心）
5. 节点资源详情按状态分层展示（按在线/告警/离线分组展示节点资源）
6. 资源告警机制（CPU/GPU/内存超阈值自动弹出告警通知 + 告警面板）
7. 多维资源动态管理（CPU/内存/GPU 利用率趋势折线图，支持按日切换）

从数据库角度看，这些功能至少需要六大类数据：

- 任务需求与调度数据
- 节点静态元数据与实时状态
- 节点与资源时序监控数据
- 超算中心与地理分布数据
- 资源告警配置与告警记录
- 聚合统计快照数据

## 3. 推荐数据库架构

建议采用 PostgreSQL 作为主库，与项目整体数据库架构一致。

连接信息：

- 数据库类型: **PostgreSQL**
- 目标库: **computing_network**
- 管理员: **cn_admin**
- 连接: `postgresql://cn_admin:slwl@10.129.209.249:5433/computing_network`

推荐逻辑分层：

- `dim_*`：维度表，存放静态或低频变化信息（节点、中心、区域等）
- `fact_*`：事实表，存放任务、告警、分配等业务事实
- `ts_*`：时序表，存放节点指标、资源趋势等高频监控数据
- `cfg_*`：配置表，存放告警阈值、系统参数等配置信息
- `vw_*`：视图或物化视图，面向接口聚合返回

## 4. 建模原则

### 4.1 复用已有表结构

本文档基于《数据库建表文档》已有的 6 张表（`supercomputing_centers`、`node_monitor`、`edge_nodes`、`tasks`、`federated_tasks`、`task_participants`）进行扩展，不重复建表，仅新增本页面特有的表。

### 4.2 请求资源与分配资源分离

任务表中同时存在 `requested`（用户请求）和 `allocated`（实际分配）两组字段，便于：
- 前端展示需求与实际对比
- 后续调度算法评估分配率
- 资源利用率统计

### 4.3 当前值与历史值分离

- 当前展示用"最新快照视图"
- 历史趋势用时序表
- 前端展示快，后端查询清晰

### 4.4 告警配置与告警记录分离

- 阈值配置存配置表，支持动态调整
- 告警记录存事实表，支持历史查询与统计分析

### 4.5 拟真数据与真实数据共存

所有关键事实表建议带上数据来源字段：

- `synthetic_realistic`
- `real_imported`
- `replayed_trace`
- `manual_patch`

后期可以逐步替换，不需要推翻表结构。

## 5. 总体实体关系

建议把整个数据库拆成 6 个主题域：

1. **任务需求域** — 任务需求提交、优先级、资源申请
2. **节点资源域** — 节点静态配置、实时状态、分层展示
3. **监控时序域** — 节点资源指标、多维趋势数据
4. **中心与地图域** — 超算中心信息、地理坐标、地图展示
5. **告警域** — 告警阈值配置、告警记录
6. **统计聚合域** — 顶部统计、资源占比、快照视图

## 7. 任务需求域

### 7.1 复用 `tasks` 表（已有）

已有表 `tasks` 可直接支撑"算力需求分层管理视图"。关键字段映射：

| 前端字段 | 数据库字段 | 说明 |
| --- | --- | --- |
| `id` | `id` | 任务唯一 ID |
| `task` | `name` | 任务名称 |
| `cpu` | `cpu_requested` | CPU 需求核数 |
| `memory` | `mem_requested` | 内存需求 (GB) |
| `gpu` | `gpu_requested` | GPU 需求张数 |
| `storage` | `duration`（折算）或新增字段 | 存储需求 |
| `priority` | `priority` | 优先级（需做值映射：`high`→`高`、`medium`→`中`、`low`→`低`） |
| `status` | `status` | 状态（需做值映射：`waiting`→`待分配`、`running`→`运行中` 等） |

**优先级值映射关系：**

| 数据库值 | 前端展示 |
| --- | --- |
| `high` | 高 |
| `medium` | 中 |
| `low` | 低 |

**状态值映射关系：**

| 数据库值 | 前端展示 |
| --- | --- |
| `waiting` | 待分配 |
| `scheduled` | 已分配 |
| `running` | 运行中 |
| `completed` | 已完成 |
| `failed` | 已失败 |
| `paused` | 已暂停 |

**建议新增字段：**

| 字段名 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| `storage_requested` | DOUBLE PRECISION | 可空 | 请求的存储容量 (GB) |
| `storage_allocated` | DOUBLE PRECISION | 可空 | 实际分配的存储容量 (GB) |

**API 设计建议：**

```sql
-- GET /api/tasks/demands
-- 查询所有任务需求，按优先级排序
SELECT
  id,
  name AS task,
  cpu_requested AS cpu,
  mem_requested AS memory,
  gpu_requested AS gpu,
  COALESCE(storage_requested, 0) AS storage,
  CASE priority
    WHEN 'high' THEN '高'
    WHEN 'medium' THEN '中'
    WHEN 'low' THEN '低'
  END AS priority,
  CASE status
    WHEN 'waiting' THEN '待分配'
    WHEN 'scheduled' THEN '已分配'
    WHEN 'running' THEN '运行中'
    WHEN 'completed' THEN '已完成'
    WHEN 'failed' THEN '已失败'
    WHEN 'paused' THEN '已暂停'
  END AS status
FROM tasks
ORDER BY
  CASE priority WHEN 'high' THEN 1 WHEN 'medium' THEN 2 WHEN 'low' THEN 3 END,
  created_at DESC;
```

### 7.2 `vw_task_stats_snapshot`（视图）

顶部统计卡片的聚合视图，后端可直接查询。

```sql
CREATE OR REPLACE VIEW vw_task_stats_snapshot AS
SELECT
  (SELECT COUNT(*) FROM tasks) AS total_tasks,
  (SELECT COUNT(*) FROM tasks WHERE status = 'running') AS running_tasks,
  (SELECT COUNT(*) FROM edge_nodes WHERE status = 'online'
    AND cpu_usage < 60 AND gpu_usage < 60) AS idle_nodes,
  (SELECT ROUND(AVG(
    (COALESCE(cpu_usage,0) + COALESCE(gpu_usage,0) + COALESCE(memory_usage,0)) / 3
  ), 1) FROM edge_nodes WHERE status = 'online') AS resource_utilization_pct;
```

**API 返回格式：**

```json
[
  { "title": "任务总数", "value": 156 },
  { "title": "运行中任务", "value": 43 },
  { "title": "空闲节点", "value": 12 },
  { "title": "资源利用率", "value": "67.3%" }
]
```

## 8. 节点资源域

### 8.1 复用 `edge_nodes` 表（已有）

已有表 `edge_nodes` 可直接支撑"节点资源详情"表格。关键字段映射：

| --- | --- | --- |

| `disk_percent` | `disk_usage` | 磁盘利用率 |

**节点状态映射关系：**

| 数据库值 | 前端展示 | 分组 |
| --- | --- | --- |
| `online` | 在线 | 🟢 在线节点 |
| `maintenance` | 告警 | 🟡 告警节点 |
| `offline` | 离线 | 🔴 离线节点 |

**建议新增字段：**

| 字段名 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| `node_code` | VARCHAR(50) | UNIQUE | 节点唯一编码，如 `node-bj-001` |
| `warning_level` | VARCHAR(20) | DEFAULT 'normal' | 告警级别：`normal` / `warning` / `critical` |

**API 设计建议：**

```sql
-- GET /api/resources/nodes
-- 查询所有节点资源详情
SELECT
  node_name,
  COALESCE(node_code, node_name) AS node_id,
  CASE status
    WHEN 'maintenance' THEN 'warning'
    ELSE status
  END AS status,
  COALESCE(cpu_usage, 0) AS cpu_percent,
  COALESCE(memory_usage, 0) AS mem_percent,
  COALESCE(gpu_usage, 0) AS gpu_percent,
  COALESCE(disk_usage, 0) AS disk_percent
FROM edge_nodes
ORDER BY
  CASE status
    WHEN 'offline' THEN 1
    WHEN 'maintenance' THEN 2
    WHEN 'online' THEN 3
  END,
  cpu_usage DESC NULLS LAST;
```

### 8.2 `vw_node_runtime_snapshot`（视图）

节点最新运行态快照视图，支撑节点详情与告警计算。

```sql
CREATE OR REPLACE VIEW vw_node_runtime_snapshot AS
SELECT
  e.node_name,
  COALESCE(e.node_code, e.node_name) AS node_id,
  e.status,
  CASE e.status
    WHEN 'maintenance' THEN 'warning'
    ELSE e.status
  END AS display_status,
  COALESCE(e.cpu_usage, 0) AS cpu_percent,
  COALESCE(e.memory_usage, 0) AS mem_percent,
  COALESCE(e.gpu_usage, 0) AS gpu_percent,
  COALESCE(e.disk_usage, 0) AS disk_percent,
  COALESCE(e.running_tasks, 0) AS running_tasks,
  e.health_score,
  e.last_heartbeat
FROM edge_nodes e;
```

### 9.2 `ts_resource_trend_5m`（新增）

5 分钟粒度全局资源趋势聚合表，前端趋势图直接查询此表，避免每次实时聚合。

| 字段名 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| `id` | BIGSERIAL | PRIMARY KEY | 自增主键 |
| `metric_time` | TIMESTAMPTZ | NOT NULL | 聚合时间点 |
| `avg_cpu_pct` | NUMERIC(5,1) | 可空 | 全局平均 CPU 利用率 |
| `avg_memory_pct` | NUMERIC(5,1) | 可空 | 全局平均内存利用率 |
| `avg_gpu_pct` | NUMERIC(5,1) | 可空 | 全局平均 GPU 利用率 |
| `node_count` | INT | 可空 | 参与聚合的节点数 |
| `source_type` | VARCHAR(30) | DEFAULT 'synthetic_realistic' | 数据来源 |

**索引：** `idx_ts_trend_time` — `(metric_time DESC)`

**生成方式：**

```sql
-- 定时从 node_monitor 聚合写入（可由定时任务或触发器完成）
INSERT INTO ts_resource_trend_5m (metric_time, avg_cpu_pct, avg_memory_pct, avg_gpu_pct, node_count)
SELECT
  date_trunc('hour', timestamp)
    + INTERVAL '5 min' * FLOOR(EXTRACT(MINUTE FROM timestamp) / 5) AS metric_time,
  ROUND(AVG(cpu_usage), 1),
  ROUND(AVG(memory_usage), 1),
  ROUND(AVG(gpu_usage), 1),
  COUNT(DISTINCT node_id)
FROM node_monitor
WHERE timestamp >= NOW() - INTERVAL '6 hours'
GROUP BY 1;
```

**API 设计建议：**

```sql
-- GET /api/resources/trend?days=7
-- 返回全局资源趋势（按日聚合 + 按时间点细查）
SELECT
  metric_time,
  avg_cpu_pct,
  avg_memory_pct,
  avg_gpu_pct
FROM ts_resource_trend_5m
WHERE metric_time >= NOW() - INTERVAL '30 days'
ORDER BY metric_time ASC;
```

**告警生成逻辑：**

```sql
-- 定时检测并写入告警记录（可由定时任务执行）
INSERT INTO fact_alert_record (node_id, node_name, metric_code, metric_name, current_value, threshold_value, alert_level, alert_message)
SELECT
  COALESCE(e.node_code, e.node_name),
  e.node_name,
  t.metric_code,
  t.metric_name,
  CASE t.metric_code
    WHEN 'cpu' THEN e.cpu_usage
    WHEN 'mem' THEN e.memory_usage
    WHEN 'gpu' THEN e.gpu_usage
    WHEN 'disk' THEN e.disk_usage
  END,
  CASE
    WHEN CASE t.metric_code
      WHEN 'cpu' THEN e.cpu_usage
      WHEN 'mem' THEN e.memory_usage
      WHEN 'gpu' THEN e.gpu_usage
      WHEN 'disk' THEN e.disk_usage
    END >= t.critical_threshold THEN t.critical_threshold
    ELSE t.warning_threshold
  END,
  CASE
    WHEN CASE t.metric_code
      WHEN 'cpu' THEN e.cpu_usage
      WHEN 'mem' THEN e.memory_usage
      WHEN 'gpu' THEN e.gpu_usage
      WHEN 'disk' THEN e.disk_usage
    END >= t.critical_threshold THEN 'critical'
    ELSE 'warning'
  END,
  e.node_name || ' ' || t.metric_name || '达 ' ||
  CASE t.metric_code
    WHEN 'cpu' THEN e.cpu_usage::text
    WHEN 'mem' THEN e.memory_usage::text
    WHEN 'gpu' THEN e.gpu_usage::text
    WHEN 'disk' THEN e.disk_usage::text
  END || t.unit || '，超过' ||
  CASE
    WHEN CASE t.metric_code
      WHEN 'cpu' THEN e.cpu_usage
      WHEN 'mem' THEN e.memory_usage
      WHEN 'gpu' THEN e.gpu_usage
      WHEN 'disk' THEN e.disk_usage
    END >= t.critical_threshold THEN '严重'
    ELSE '警告'
  END || '阈值 ' ||
  CASE
    WHEN CASE t.metric_code
      WHEN 'cpu' THEN e.cpu_usage
      WHEN 'mem' THEN e.memory_usage
      WHEN 'gpu' THEN e.gpu_usage
      WHEN 'disk' THEN e.disk_usage
    END >= t.critical_threshold THEN t.critical_threshold::text
    ELSE t.warning_threshold::text
  END || t.unit
FROM edge_nodes e
CROSS JOIN cfg_alert_threshold t
WHERE t.is_enabled = true
  AND e.status = 'online'
  AND CASE t.metric_code
    WHEN 'cpu' THEN e.cpu_usage
    WHEN 'mem' THEN e.memory_usage
    WHEN 'gpu' THEN e.gpu_usage
    WHEN 'disk' THEN e.disk_usage
  END >= t.warning_threshold;
```
