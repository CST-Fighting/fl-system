# 算力任务需求管理模块交接说明

## 1. 我负责的部分

我负责的是算力网络智能调度系统中的"算力任务需求管理"数据底座与前后端数据契约设计。

这部分对应页面为：

- `src/pages/ComputingSchedule/TaskManagement/index.tsx`

它不是单纯的前端展示页，而是连接以下几类能力的核心模块：

- 顶部统计概览：任务总数、运行中任务、空闲节点、资源利用率
- 算力需求分层管理视图：按高/中/低优先级分组展示任务需求
- 算力中心能力视图：CPU/GPU/存储/网络资源占比环形饼图
- 全国算力节点分布：中国地图散点图，展示 26 个算力中心
- 节点资源详情按状态分层展示：按在线/告警/离线分组展示节点资源
- 资源告警机制：CPU/GPU/内存超阈值自动弹出告警通知 + 告警面板
- 多维资源动态管理：CPU/内存/GPU 利用率趋势折线图，支持按日切换

简单说，这部分的职责是：让前端"算力任务需求管理"页面背后的数据结构、接口、数据库表、拟真数据和未来真实数据接入方式都能对齐。

## 3. 当前数据流

当前系统的数据流是：

1. 任务需求数据：FastAPI 路由返回硬编码字典 → 前端 `getTaskDemands()` 请求 → 页面渲染任务表格
2. 统计卡片数据：FastAPI 路由返回硬编码字典 → 前端 `getTaskStats()` 请求 → 页面渲染统计卡片
3. 节点资源数据：CSV 文件 → 前端 hooks 解析 → 页面渲染节点表格、状态分组、饼图
4. 趋势数据：CSV 文件 → 前端 hooks 解析 → 页面渲染趋势折线图
5. 地图数据：hooks 中硬编码常量 → 页面渲染地图散点
6. 告警数据：前端根据节点数据和阈值配置实时计算 → 弹出通知

当前还没有把数据库接入进来。数据库设计已经完成，下一步工作是把 CSV/硬编码/mock 数据逐步替换为 DB-backed API。

## 4. 数据库设计核心思路

数据库使用 PostgreSQL，与项目整体数据库架构一致：

- 连接地址：`postgresql://cn_admin:slwl@10.129.209.249:5433/computing_network`

表命名分为五类：

- `dim_*`：维度表，存放静态或低频变化信息（节点、中心、区域等）
- `fact_*`：事实表，存放任务、告警、分配等业务事实
- `ts_*`：时序表，存放节点指标、资源趋势等高频监控数据
- `cfg_*`：配置表，存放告警阈值、系统参数等配置信息
- `vw_*`：视图或物化视图，面向接口聚合返回

最关键的建模原则：

1. **复用已有表结构**：基于 `数据库建表文档.md` 已有的 6 张表扩展，不重复建表
2. **请求资源与分配资源分离**：任务表中同时存在 `requested`（用户请求）和 `allocated`（实际分配）两组字段
3. **当前值与历史值分离**：当前展示用快照视图，历史趋势用时序表
4. **告警配置与告警记录分离**：阈值配置存配置表，告警记录存事实表
5. **拟真数据与真实数据共存**：关键事实表带 `source_type` 字段标识数据来源

## 7. 建议新增/改造接口

为了让页面完全由后端/数据库驱动，建议新增或改造这些接口：

### 已有接口（需改造为数据库查询）

- `GET /api/tasks/demands` — 改造为从 `tasks` 表查询，带优先级值映射
- `GET /api/tasks/stats` — 改造为从 `vw_task_stats_snapshot` 视图查询

### 建议新增接口

- `GET /api/resources/nodes` — 节点资源详情列表（从 `edge_nodes` + `vw_node_runtime_snapshot`）
- `GET /api/resources/usage` — 资源占比数据（从后端聚合或 `vw_resource_usage_snapshot`）
- `GET /api/resources/map` — 全国算力节点分布（从 `dim_computing_center_geo`）
- `GET /api/resources/trend?days=7` — 资源趋势数据（从 `ts_resource_trend_5m`）
- `GET /api/alerts/active` — 当前活跃告警列表（从 `fact_alert_record`）
- `POST /api/alerts/{id}/acknowledge` — 确认告警
- `GET /api/alerts/thresholds` — 获取告警阈值配置
- `PUT /api/alerts/thresholds/{metric_code}` — 修改告警阈值

这些接口补齐后，页面中的 CSV 解析、硬编码常量和前端本地告警计算就可以逐步删除。

## 9. 接手人需要特别注意的点

### 9.1 不要随意改前端字段名

`src/pages/ComputingSchedule/TaskManagement/types.ts` 里的 TypeScript 类型就是当前前端契约。后端迁移到数据库时，最好保持响应字段兼容。

如果确实要改字段名，应同时修改：

- 前端类型定义（`types.ts`）
- 页面使用字段（`index.tsx`）
- 数据管理 Hook（`hooks.ts`）
- 后端响应（`tasks.py`）
- 相关文档

### 9.2 前端 CSV 和后端接口的数据不要混用

当前 `hooks.ts` 中：

- 任务需求和统计卡片来自后端接口（`getTaskDemands`、`getTaskStats`）
- 节点资源和趋势数据来自前端 CSV（`parseNodeCSV`、`parseTrendCSV`）

迁移时不要把 CSV 数据直接搬到后端接口然后删掉前端解析逻辑，应该分步进行：

1. 先把 CSV 数据导入数据库
2. 再新增后端接口从数据库查询
3. 最后改前端 hooks 从接口获取

### 9.3 优先级和状态值需要做映射

数据库中优先级存储的是 `high`/`medium`/`low`，前端展示的是 `高`/`中`/`低`。

数据库中状态存储的是 `waiting`/`running`/`completed` 等，前端展示的是 `待分配`/`运行中`/`已完成` 等。

后端接口应该做值映射（SQL CASE 或 Python 层面），保持前端不需要改。

### 9.4 节点状态映射关系

数据库 `edge_nodes.status` 与前端展示的对应关系：

| 数据库值 | 前端展示 | 分组 |
| --- | --- | --- |
| `online` | 在线 | 🟢 在线节点 |
| `maintenance` | 告警 | 🟡 告警节点 |
| `offline` | 离线 | 🔴 离线节点 |

注意前端代码中 `maintenance` 映射为 `warning`，接手人在写后端接口时需要保持这个映射。

### 9.5 告警阈值与前端一致

当前前端 `types.ts` 中的 `ALERT_THRESHOLDS` 定义了告警阈值：

- CPU 利用率：警告 60%，严重 80%
- 内存利用率：警告 60%，严重 80%
- GPU 利用率：警告 60%，严重 80%

后端 `cfg_alert_threshold` 表的预置数据应与这些值一致，迁移后前端可以从后端获取阈值配置，而不是硬编码。

### 9.6 趋势数据的日概览和日明细

前端趋势图有两层结构：

- `dailyOverview`：每日平均值概览折线图
- `dailyDetailMap`：点击某一天后展示当天 5 分钟粒度明细

数据库设计中的 `ts_resource_trend_5m` 只存 5 分钟粒度数据，日概览应由后端聚合返回，或者建一个日级聚合视图。

### 9.7 地图数据需要稳定

26 个算力中心的经纬度坐标、名称、容量、层级应存入 `dim_computing_center_geo` 表。后续新增或修改算力中心时，只改数据库，不需要改前端代码。

### 9.8 数据来源字段要保留

关键表建议保留 `source_type` 字段，例如：

- `synthetic_realistic`
- `real_imported`
- `replayed_trace`
- `manual_patch`

这样未来真实数据和拟真数据可以共存，迁移时不会混乱。
