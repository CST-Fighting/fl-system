# 协同预测预分配模块交接说明

## 1. 我负责的部分

我负责的是算力网络智能调度系统中的“协同预测与预分配”数据底座与前后端数据契约设计。

这部分对应页面为：

- `src/pages/ComputingSchedule/PredictionAllocation/index.tsx`

它不是单纯的前端展示页，而是连接以下几类能力的核心模块：

- 多层级算力视角切换：全局、区域、省级、单节点
- 节点资源监控：CPU、内存、GPU、磁盘、带宽、时延、抖动、丢包
- 资源需求预测：日级、月级、短时节点负载预测
- 调度预分配：任务进入系统后，根据当前资源和预测负载选择目标节点
- 拓扑展示：国家调度中心、枢纽、区域中心、省级算网大脑、DC 节点、边缘节点
- 调度过程解释：候选节点评分、任务分配、重定向、调度日志
- 后续数据库落地：把当前 JSON/mock 数据平滑迁移到 PostgreSQL/TimescaleDB

简单说，这部分的职责是：让前端“协同预测与分配”页面背后的数据结构、接口、数据库表、拟真数据和未来真实数据接入方式都能对齐。

## 3. 当前数据流

当前系统的数据流是：

1. 数据生成脚本生成 JSON
2. JSON 存在 `server/app/data/prediction_allocation/`
3. FastAPI 通过 `prediction_allocation_data.py` 读取 JSON
4. 后端路由返回接口数据
5. 前端通过 `predictionAllocation.ts` 请求接口
6. 页面渲染预测曲线、分配表、节点监控和流量图

当前还没有真正把数据库接入进来。数据库设计已经完成，下一步工作是把 JSON-backed API 逐步替换为 DB-backed API。

## 4. 数据库设计核心思路

数据库推荐使用：

- PostgreSQL：存节点、任务、拓扑、调度、策略等关系型数据
- TimescaleDB：存节点监控、GPU 监控、链路监控、预测序列等时序数据

表命名分为四类：

- `dim_*`：维度表，存静态或低频变化数据
- `fact_*`：事实表，存任务、调度、日志、评估等业务事实
- `ts_*`：时序表，存监控、预测、链路等高频数据
- `vw_*`：视图或物化视图，面向接口聚合返回

最关键的建模原则是：拓扑节点和算力节点不要混成一张简单节点表。

页面里的“节点”分两类：

- 控制/服务节点：国家级调度中心、枢纽、区域中心、省级算网大脑、预测引擎、策略代理、网络编排
- 算力节点：DC 节点、边缘节点、资源池节点

所以设计中使用：

- `dim_topology_vertex` 统一描述拓扑图上的所有点
- `dim_compute_node` 专门描述可承载任务的算力节点
- `dim_topology_vertex.compute_node_id` 在需要时关联真实算力节点

## 8. 后续开发顺序建议

### 第一步：先保接口，不动前端

先把当前 JSON 数据导入数据库，然后改 `prediction_allocation_data.py`，让现有接口从数据库读取，但返回字段保持不变。

优先替换这些接口：

- `/api/allocation/nodes`
- `/api/allocation/nodes/overview`
- `/api/allocation/nodes/{node_id}/dashboard`
- `/api/allocation/nodes/{node_id}/history`

这样风险最低，因为前端页面结构不需要变。

### 第二步：接入任务与调度数据

落库并接入：

- `fact_task`
- `fact_task_requirement`
- `fact_task_assignment`
- `fact_schedule_log`

然后把页面中的任务队列和调度日志改为接口返回。

### 第三步：接入拓扑数据

落库并接入：

- `dim_topology_vertex`
- `dim_topology_edge`
- `dim_topology_layout`
- `fact_topology_runtime_state`

然后新增 `/api/allocation/topology/view` 接口，让拓扑节点、边、状态、布局坐标全部来自数据库。

### 第四步：接入预测与重定向

落库并接入：

- `fact_forecast_run`
- `ts_forecast_point`
- `ts_node_short_forecast_snapshot`
- `fact_reroute_event`

这样页面可以展示真实预测曲线、预测负载环、预测重定向边和预测触发的调度决策。

### 第五步：接入真实数据源

真实数据接入建议按这个顺序替换：

1. 替换节点静态信息：`dim_compute_node`、`dim_node_gpu_device`
2. 替换监控时序：`ts_node_resource_metric_1m`、`ts_gpu_resource_metric_1m`
3. 替换任务调度：`fact_task`、`fact_task_assignment`、`fact_schedule_log`
4. 替换预测结果：`fact_forecast_run`、`ts_forecast_point`

不要一次性全部替换，否则很难判断问题来自数据、接口还是前端。

## 9. 接手人需要特别注意的点

### 9.1 不要随意改前端字段名

`src/services/api/predictionAllocation.ts` 里的 TypeScript 类型就是当前前端契约。后端迁移到数据库时，最好保持响应字段兼容。

如果确实要改字段名，应同时修改：

- 前端类型定义
- 页面使用字段
- 后端响应
- 测试或示例 JSON
- 相关文档

### 9.2 拓扑布局要稳定

页面拓扑不适合每次刷新重新随机布局。数据库中设计了 `dim_topology_layout`，就是为了保存每个视角下的节点坐标。

后续接口应返回稳定坐标，避免前端刷新后节点跳动。

### 9.3 当前负载和预测负载要分开

拓扑图中：

- 实心内圈表示当前真实负载
- 虚线外圈表示未来 10 分钟预测负载

数据库中对应：

- 当前值：`ts_node_resource_metric_1m` 或 `vw_node_runtime_snapshot`
- 预测值：`ts_node_short_forecast_snapshot`

不要只用一个 CPU 字段同时充当当前值和预测值。

### 9.4 拟真数据也要有因果关系

拟真数据不应只是随机数。推荐保持这些关系：

- CPU 高时，时延、抖动、丢包上升
- 高资源任务更倾向分配到能力强的节点
- 节点离线会触发调度日志和重定向事件
- 新节点接入会影响候选节点评分和负载均衡
- 预测到热点后，提前出现重定向边或调度策略变化

这样前端展示才像一个真实系统，而不是静态看板。

### 9.5 数据来源字段要保留

关键表建议保留 `source_type` 字段，例如：

- `synthetic_realistic`
- `real_imported`
- `trace_replay`
- `manual_patch`

这样未来真实数据和拟真数据可以共存，迁移时不会混乱。

## 10. 验收标准

接手人完成后，可以按以下标准判断这部分是否迁移成功：

- 前端页面能正常打开，没有字段缺失报错
- 现有 10 个接口仍能返回兼容数据
- 节点列表、节点详情、节点历史曲线来自数据库
- 任务分配结果来自数据库
- 调度日志可以按节点或拓扑顶点查询
- 拓扑图节点和边来自数据库，布局稳定
- 当前负载和预测负载能同时展示
- 重新生成或导入数据后，不需要改前端页面代码
- 文档中的接口、表名、字段名与实际代码一致
