# 基于强化联邦学习的算力网络系统 — 统一数据库设计

> 本文档整合项目中所有功能模块的数据需求（边缘控制层、云中心控制层、用户控制层、算力任务管理、协同预测预分配），建立全项目统一的 PostgreSQL 数据库。

---

## 一、设计概述

### 1.1 目标

- 消除各模块间的字段重复、命名不一致、数据冗余
- 统一命名规范（snake_case）、数据类型和约束
- 建立合理的主外键、索引，支撑所有前端页面和后端服务
- 时序数据可平滑迁移至 TimescaleDB

### 1.2 目标数据库

- **数据库类型**: PostgreSQL（建议 ≥ 14），时序场景可选 TimescaleDB 扩展
- **数据库名称**: `computing_network`
- **连接**: `postgresql://cn_admin:<password>@<host>:5433/computing_network`

### 1.3 命名规范

| 分类 | 前缀 | 说明 | 示例 |
|------|------|------|------|
| 维度表 | `dim_` | 静态或低频变化数据 | `dim_region` |
| 事实表 | `fact_` | 业务事实记录 | `fact_task` |
| 时序表 | `ts_` | 高频监控/预测序列 | `ts_node_metric` |
| 配置表 | `cfg_` | 系统配置参数 | `cfg_alert_threshold` |
| 关系表 | `rel_` | 多对多关联 | `rel_pool_node` |
| 视图 | `vw_` | 聚合快照视图 | `vw_node_runtime_snapshot` |

字段命名统一 snake_case，时间字段用 `*_at`（如 `created_at`），百分比字段用 `*_pct`，标识字段用 `*_id`。

### 1.4 设计原则

1. **维度与事实分离** — 静态元数据放维度表，业务记录放事实表
2. **当前快照与历史时序分离** — 当前展示用视图/快照，趋势用时序表
3. **请求资源与分配资源分离** — 便于评估调度效果
4. **拟真数据与真实数据兼容** — 关键表带 `source_type` 字段，后期可平滑替换

## 四、视图设计

### `vw_task_stats_snapshot`
> **负责人：陈舒婷**

任务统计快照，支撑顶部统计卡片。

```sql
CREATE OR REPLACE VIEW vw_task_stats_snapshot AS
SELECT
  (SELECT COUNT(*) FROM fact_task) AS total_tasks,
  (SELECT COUNT(*) FROM fact_task WHERE status = 'running') AS running_tasks,
  (SELECT COUNT(*) FROM dim_compute_node WHERE status = 'online'
    AND EXISTS (
      SELECT 1 FROM ts_node_metric m
      WHERE m.node_id = dim_compute_node.node_id
      ORDER BY m.metric_time DESC LIMIT 1
      HAVING m.cpu_usage_pct < 60 AND m.gpu_usage_pct < 60
    )) AS idle_nodes,
  (SELECT ROUND(AVG(
    (COALESCE(cpu_usage_pct,0) + COALESCE(gpu_usage_pct,0) + COALESCE(memory_usage_pct,0)) / 3
  ), 1) FROM ts_node_metric WHERE node_type = 'edge'
    AND metric_time = (SELECT MAX(metric_time) FROM ts_node_metric)
  ) AS resource_utilization_pct;
```

### `vw_node_runtime_snapshot`
> **负责人：陈舒婷、张鼎功**

节点最新运行态快照。

```sql
CREATE OR REPLACE VIEW vw_node_runtime_snapshot AS
SELECT
  n.node_id,
  n.node_name,
  COALESCE(n.node_code, n.node_name) AS node_code,
  n.status,
  CASE n.status
    WHEN 'maintenance' THEN 'warning'
    ELSE n.status
  END AS display_status,
  n.location,
  COALESCE(m.cpu_usage_pct, 0) AS cpu_percent,
  COALESCE(m.memory_usage_pct, 0) AS mem_percent,
  COALESCE(m.gpu_usage_pct, 0) AS gpu_percent,
  COALESCE(m.disk_usage_pct, 0) AS disk_percent,
  COALESCE(n.running_tasks, 0) AS running_tasks,
  n.health_score,
  n.last_heartbeat,
  m.metric_time AS latest_metric_time
FROM dim_compute_node n
LEFT JOIN LATERAL (
  SELECT cpu_usage_pct, memory_usage_pct, gpu_usage_pct, disk_usage_pct, metric_time
  FROM ts_node_metric
  WHERE node_id = n.node_id
  ORDER BY metric_time DESC LIMIT 1
) m ON TRUE;
```

### `vw_resource_usage_snapshot`
> **负责人：陈舒婷**

资源占比快照，支撑饼图。

```sql
CREATE OR REPLACE VIEW vw_resource_usage_snapshot AS
SELECT 'CPU' AS name, ROUND(AVG(COALESCE(cpu_usage_pct, 0)), 1) AS value
  FROM vw_node_runtime_snapshot WHERE status = 'online'
UNION ALL
SELECT 'GPU' AS name, ROUND(AVG(COALESCE(gpu_percent, 0)), 1) AS value
  FROM vw_node_runtime_snapshot WHERE status = 'online'
UNION ALL
SELECT '存储' AS name, ROUND(AVG(COALESCE(disk_percent, 0)), 1) AS value
  FROM vw_node_runtime_snapshot WHERE status = 'online'
UNION ALL
SELECT '网络' AS name,
  ROUND(AVG(COALESCE(network_bandwidth_mbps, 0)) / 10.0, 1) AS value
  FROM dim_compute_node WHERE status = 'online' AND network_bandwidth_mbps IS NOT NULL;
```

```sql
  NOW() AS snapshot_time,
  COALESCE(ROUND(AVG(latency_ms), 1), 0) AS avg_delay_ms,
  COALESCE(ROUND(AVG(ROUND(STDDEV(cpu_usage_pct), 1)), 1), 0) AS load_std,
  (SELECT COUNT(*) FROM fact_task WHERE status = 'completed') AS success_tasks,
  COALESCE(ROUND(AVG(bandwidth_usage_gbps), 2), 0) AS avg_bandwidth_gbps,
  COALESCE(ROUND(AVG(gpu_usage_pct), 1), 0) AS avg_gpu_usage_pct
FROM vw_node_runtime_snapshot
WHERE status = 'online';
```

### 7.2 TimescaleDB 建议

对高频写入的时序表（`ts_node_metric`、`ts_gpu_metric`），建议：
1. 启用 TimescaleDB 扩展
2. 将表转为 hypertable
3. 设置压缩策略（7天后压缩）
4. 设置保留策略（自动删除过期数据）

```sql
-- 示例：ts_node_metric
SELECT create_hypertable('ts_node_metric', 'metric_time', chunk_time_interval => INTERVAL '1 day');
ALTER TABLE ts_node_metric SET (timescaledb.compress, timescaledb.compress_segmentby = 'node_id');
SELECT add_compression_policy('ts_node_metric', INTERVAL '7 days');
SELECT add_retention_policy('ts_node_metric', INTERVAL '90 days');
```

## 十一、总结

本统一数据库设计将项目中 **4 份独立数据文档**中的实体合并为 **44 张表 + 4 个视图**，覆盖：

| 域 | 表数 | 核心能力 |
|----|:---:|---------|
| 区域与组织 | 1 | 多级区域、视角切换 |
| 算力节点 | 5 | 超算中心、边缘节点、GPU、资源池 |
| 拓扑 | 6 | 统一顶点建模、布局、运行态、事件 |
| 客户端 | 1 | 客户端归属 |
| 用户 | 1 | 用户认证 |
| 任务 | 5 | 任务主表、需求、分配、执行、评分 |
| 联邦学习 | 3 | FL扩展、参与者、训练轮次 |
| 监控时序 | 4 | 节点/GPU/磁盘/趋势聚合 |
| 预测 | 3 | 预测运行、序列点、快照 |
| 调度 | 2 | 调度日志、重定向 |
| 告警 | 2 | 阈值配置、告警记录 |
| 流量链路 | 3 | 链路监控、流量快照、时序 |
| 策略评估 | 3 | 策略字典、评估运行、指标 |
| 数据接入 | 3 | 数据源、批次、仿真场景 |

**核心设计特点：**

1. **统一拓扑建模** — 控制节点、算力节点、客户端统一用 `dim_topology_vertex`，适合所有页面拓扑展示
2. **维度与事实分离** — 静态元数据放维度表，监控指标放时序表，业务记录放事实表
3. **拟真与真实兼容** — 关键表带 `source_type` 字段，先用拟真数据跑通，后续平滑替换为真实数据
4. **三阶段渐进落地** — 16 → 30 → 44 张表，优先支撑核心页面，逐步补全高级功能
